1 The History of MDRevealer

MDRevealer’s origin lies in a series of Perl scripts written by Robert Clifford from the Multidrug resistant organism Repository and Surveillance Network (MRSN) at the Walter Reed Army Institute of Research (WRAIR). For hospitals, manually parsing the results from identification / antibiotic susceptibility testing (ID/AST) machines to identify multidrug resistant isolates was time-consuming and error-prone. Therefore, he wrote a set of Perl scripts to automate this process.

Unfortunately, these Perl scripts had their own shortcomings. For one, they only ran on computers which had Perl installed; for another, they lacked a user-friendly interface. In order to resolve these issues, Robin Clifford, while an intern at WRAIR, re-implemented them as a Java application. He addded a graphical user interface (GUI), localization support via GNU gettext, and extended the functionality of the program to make it easier for users to customize definitions of multidrug resistant and antibiotic susceptiblity.

MDRevealer processes files from Phoenix, Vitek2, and Microscan ID/AST machines, and produces output in multiple formats.

2 Configuring MDRevealer

Before using MDRevealer, you must first set up its configuration files. The main configuration file is located in the "data" directory of the program, and is called config.txt. By default, it looks like this: 



Facility     0000
Machine name     Test Machine

#If you want the program to write its output files to the same place every
#time, delete the # marks from the lines below and change the names of
#"report.txt" and so on to what you want the outputs to be called.

Default detailed report location     output/report.txt
Default summary location     output/summary.txt
Default spreadsheet location     output/spreadsheet.txt
Default .dat location     output/dat.txt
Default localized output     output/localized.txt

Default MDR file     data/mdr.txt
Default breakpoints file     data/breakpoints.txt
Custom parser file     data/parsers/my_parser.txt

.dat output
lastName     firstName     middleI     dmisId     spin     
ssnPrefix     ssn     dob     gender     accession      
isolationDate     cultureType     source     locationType     
location     isolate     organism     alternateOrganism     
equipment     antibiotic     mic     call


The contents of config.txt are described in the following sections.

2.1 The "Facility" Field

MDRevealer looks for a line beginning with "Facility" (minus the quotes), a tab, and the four-digit ID number of the facility the data came from. The ID number refers to a facility in data/hospitals.txt. The hospitals.txt file included with the program contains a list of all US Army medical facilities. To add a new hospital, the entry for ID number 0000 can be edited.

2.2 The "Machine name" Field

MDRevealer looks for a line beginning with "Machine name" (minus the quotes), a tab, and the name of the machine the isolates were tested on. This lets the user specify the equipment name if it isn’t provided in the input file.

If a parser isn’t uncommented in config.txt, then MDRevealer will display a warning and close.

2.4 Default Locations

Each type of MDRevealer output file has a default name and location. If you don’t want the program to automatically write your output files to a default location, simply delete those lines. Otherwise, change "report.txt", "summary.txt", "spreadsheet.txt", etc. to your preferred location. All locations are relative to the program (MDRevealer.jar). For example, if you want the program to write your detailed reports to "report.txt" in a folder called "output", which is in the same folder as the program, you would do this: 

...
Default detailed report location     output/report.txt
...

If you want it to be written to a folder called "output" in your home directory, you would do this (on GNU/Linux, *BSD, OS X, etc.): 

Default detailed report location     /home/[your username]/output/report.txt

Or on Windows: 

Default detailed report location     C:/Users/[your username]/output/report.txt

Please note that the directory must be created before MDRevealer is run.

2.5 Default MDR Criteria and Drug Susceptibility Breakpoints

If you want to use custom antibiotic susceptibility breakpoints or MDR criteria, just comment out the lines, and when you run the program, it’ll prompt you to give it an MDR definition file and a breakpoint file every time. To comment them out, change the lines to this: 

#Default MDR file     data/mdr.txt
#Default breakpoints file     data/breakpoints.txt

2.6 Custom parser file

This line points to the file that will be used when the user selects the "custom parser" option in MDRevealer.

2.7 .dat output.

If the program finds a line in config.txt labeled ".dat output", it uses the next line to determine which data fields should appear in the tab-delimited .dat output file.

The fields the program recognizes are:

  • lastName = The last name of the patient. (Ex: SMITH)
  • firstName = The first name of the patient. (Ex: JOHN)
  • middleI = The middle initial of the patient. (Ex: Q)
  • dmisId = The facility the bacterium was found at. (Ex: 9009 TEST HOSPITAL)
  • spin = An identifier used by MRSN.
  • ssnPrefix = The SSN prefix of the patient, used in the US military healthcare system. (Ex: 20)
  • ssn = The Social Security Number (SSN) of the patient. (Ex: 987654320)
  • dob = The birth date of the patient. (Ex: 07/04/1776)
  • gender = The gender of the patient. (Ex: F)
  • accession = The accesssion number of the bacterium. (Ex: 12121)
  • isolationDate = The date the clinical sample was taken from the patient. (Ex: 7/16/2013)
  • cultureType = A general description of the tissue or fluid that the organism was isolated from. (Ex: BLOOD)
  • source = A more specific version of cultureType. (Ex: LEG ABSCESS)
  • locationType = A general hospital location. (Ex: ICU)
  • location = A specific location within a hospital. (Ex: WARD 5)
  • isolate = The isolate number of the bacterium. If multiple species of bacterium are isolated from a single culture, each has the same accession, but a unique isolate number. (Ex: 2)
  • organism = The organism’s display name from organisms.txt. (Ex: E. coli)
  • alternateOrganism = Rarely used; generally only found when "organism" is "other". (Ex: Shigella sonnei)
  • longOrganism = The organism’s full name. (Ex: Acinetobacter baumannii)
  • equipment = The ID/AST equipment used to produce the data. (Ex: Vitek2)
  • antibiotic = The name of the drug tested. Each drug tested will appear on a separate row. (Ex: Amikacin)
  • mic = The minimum inhibitory concentration. Each MIC will appear on a separate row. (Ex: <=1)
  • call = Whether the organism was sensitive, resistant, or indeterminate. Each call will appear on a separate row. (Ex: S)

The user can delete or reorder these fields in config.txt as they see fit in order to create their own customized .dat file.

3 Configuring the XML Tests

MDRevealer 2.0 added support for writing your own tests for carbapenem resistance, MRSA status, VRE status, or ESBL status. An example: 

<?xml version="1.0"?>
	<test>
		<test_name>PSA Carb Test</test_name>
		<bacterium>Pseudomonas</bacterium>
		<condition number = "1" default = "passed">
			<drug>
				<name>Cefepime</name>
				<if read_type = "Call" value = "S">
					<condition_status>failed</condition_status>
				</if>
			</drug>
			<drug>
				<name>Cefotaxime</name>
				<if read_type = "Call" value = "S">
					<condition_status>failed</condition_status>
				</if>
			</drug>
			<drug>
				<name>Ceftazidime</name>
				<if read_type = "Call" value = "S">
					<condition_status>failed</condition_status>
				</if>
			</drug>
			<drug>
				<name>Ceftriaxone</name>
				<if read_type = "Call" value = "S">
					<condition_status>failed</condition_status>
				</if>
			</drug>
			<drug>
				<name>Ertapenem</name>
				<if read_type = "Call" value = "S">
					<condition_status>failed</condition_status>
				</if>
			</drug>
		</condition>
		<condition number = "2" default = "failed" number_required = "1">
			<drug>
				<name>Doripenem</name>
				<if read_type = "MIC" comparison="greater_than_equal" value = "4">
					<condition_status>+</condition_status>
				</if>
				<name>Imipenem</name>
				<if read_type = "MIC" comparison="greater_than_equal" value = "4">
					<condition_status>+</condition_status>
				</if>
				<name>Meropenem</name>
				<if read_type = "MIC" comparison="greater_than_equal" value = "4">
					<condition_status>+</condition_status>
				</if>
			</drug>
		</condition>
		<required_condition>1</required_condition>
		<required_condition>2</required_condition>
		<result>Carb</result>
	</test>

In English, this works out to... 

Test all bacteria with Pseudomonas in their name.
Condition #1 is met by default, and must be failed.
If the call of Cefepime is S, fail the condition.
If the call of Cefotaxime is S, fail the condition.
If the call of Ceftazidime is S, fail the condition.
If the call of Ceftriaxone is S, fail the condition.
If the call of Ertapenem is S, fail the condition.
Condition #2 is failed by default, and must be met.
If one sub-condition is true, the condition is met.
If the MIC of Doripenem is >=4, the sub-condition is met.
If the MIC of Imipenem is >=4, the sub-condition is met.
If the MIC of Meropenem is >=4, the sub-condition is met.
Condition 1 must be met.
Condition 2 must be met.
If the above conditions are met, the bacterium is carbapenem resistant.

Here’s a list of tags MDRevealer will recognize.

  • <test>: Defines a test.
  • <test_name>: Defines the name of the test. It gets printed to stdout when the program executes the test.
  • <bacterium>: The name of the bacterium to run tests against. The program will look for any bacteria containing this name: <bacterium>Klebsiella</bacterium> will result in the program running its tests against any bacterium with "Klebsiella" in its name. Setting the name to "Generic" will have it run tests on any bacteria whose name is not found in "carbapenem.txt". Setting the name to "Any" will have it run tests on all bacteria.
  • <condition>: Defines a condition.
    • Attributes:
    • number: The number of the condition, used for identifying it.
    • default: Must be either "passed" or "failed". It determines if the test is considered to be passed or failed by default.
    • number_required: The number of sub-conditions that must be passed to consider the condition passed.

      For example: 

      
For condition 2: one or more of Doripenem, Imipenem, and Meropnem must
have a MIC >=4.

In this case, Doripenem's MIC being >=4 would be a sub-condition.
  • <drug>: Defines a drug.
  • <name>: Defines the name of a drug.
  • <if>: If the conditions in its attributes are met, the program does whatever <condition_status> says to do.
    • Attributes:
    • read_type: The type of read to look at: call or MIC.
    • comparison: Must be either "less_than_equal" (<=), "less_than" (<), "greater_than_equal" (>=), or "greater_than" (>).
    • value: What value to compare the read_type to.
  • <condition_status>: What to do if <if> is met.
    • failed: Fail the condition.
    • passed: Pass the condition.
    • +: Pass the sub-condition.
    • -: Fail the sub-condition.
  • <required_condition>: This condition must be met to pass the test.
  • <required_if>: This condition must be met to pass the test, but only if the requirements in its attributes are met.

    An example: 

    		<required_if drug="Ertapenem" read_type="MIC" comparison="greater_than_equal" value="1">1</required_if>
		<required_if drug="Ertapenem" read_type="MIC" comparison="less_than_equal" value="0">2</required_if>
		<required_if drug="Ertapenem" read_type="MIC" comparison="less_than_equal" value="0.5">3</required_if>

    And in English: 

    If the MIC of Ertapenem is >=1, condition 1 must be met.
If the MIC of Ertapenem is <=0, condition 2 must be met.
If the MIC of Ertapenem is <=0.5, condition 3 must be met.

    Attributes are the same as the ones for <if>.

  • <result>: What to do if the test is passed.
    • Carb: Mark the isolate as carbapenem resistant.
    • MRSA: Mark the isolate as MRSA.
    • ESBL: Mark the isolate as ESBL.
    • VRE: Mark the isolate as VRE.
    • Anything else: Mark the isolate as whatever you put between the tags.

5 Other Configuration Files

Each data file used by MDRevealer has one or more fields, each separated by a tab.

5.1 breakpoints.txt

This file is used to provide customized susceptibility breakpoints that will replace the susceptibility calls made by an ID/AST machine.

A line in breakpoints.txt looks like this: 

ACB     Amikacin     16     64

The first column is the name of the organism.

The second column is the name of the drug you’re testing against the organism.

If the MIC of the drug is less than or equal to the value in the third column, the program will call it sensitive (S).

If the MIC of the drug is greater than or equal to the value in the fourth column, the program will call it resistant (R).

If the MIC of the drug is in between the two values, the program will call it indeterminate (I).

If a breakpoints file is selected, and the input file contains an organism/drug combination absent from mdr.txt, the program will write a warning and set the call to "X".

5.2 carbapenem.txt

A line in carbapenem.txt looks like this: 

ACB

Any organism whose name is found in this file will be excluded from the generic carbapenem test.

All names in carbapenem.txt must be in ALL CAPS.

5.3 drugnames.txt

A line in drugnames.txt looks like this: 

amikacin     Amikacin

The first column is the synonym for a drug.

The second column is the preferred drug name.

This file helps prevent confusion due to alternate names for drugs, or simple misspellings. To avoid false positives in warnings.txt, the proper name must also be in the file, like so: 

Amikacin     Amikacin

5.4 hospitals.txt

A line in hospitals.txt looks like this: 

0001     FOX AHC-REDSTONE ARSENAL

The first column is the facility’s ID number. The program uses this field for creating the SPIN, an identifier for each bacterium used by MRSN.

The second column is the name of the facility.

5.5 mdr.txt

A line in mdr.txt looks like this: 

ACB     A     Cefepime

The first column is the organism’s name.

The second column is the category of the drug. Each drug is assigned to an antibiotic category, such as beta-lactams, aminoglycosides, fluoroquinolones, etc. Each category should be marked with a letter: A through O. You shouldn’t need more than 15 categories, but if you do, email me. See Contact.

The last column is the organism’s name.

5.6 organisms.txt

A line in organisms.txt looks like this: 

ACINETOBACTER LOWFFI     Acinetobacter lwoffii     A. lwoffii     834683456

The first column is a common name. This name must be in ALL CAPS.

The second name is used internally by the program. It must be genus and species.

The last name is the display name used in the output files. It can be whatever the user wants, but by default, it’s the first letter of the genus, a dot, a space, and the species.

The number at the end is the organism’s SNOMED code. Currently the program doesn’t use this for anything, but it might at some point. This is optional, so feel free to leave it off.

5.7 Submitting changes

If you add anything to any of these config files, please contact me at rsclifford(at)hotmail(dot)com, so I can add it to the master data file.

6 Using MDRevealer

Before running the program, make sure that every organism name in your input file is in organisms.txt, and if not, add it. Otherwise, you won’t get the right results.

Once you’ve got the program configured, it’s time to actually run it. A brief explanation of each option follows.

  • Detailed report (in English) - Prints out a detailed list of all drugs a bacterium’s resistant/sensitive to.
  • Summary (in English) - Prints accession, isolate, organism name, and MDR status.
  • Spreadsheet (in English) - Prints out a tab-delimited text file with one sample per row.
  • .dat file (in English) - Prints out a tab-delimited text file with one organism/ drug per row.
  • Detailed report (in your current language) - Prints out a detailed report in the OS’s current language.
  • Ignore calls from the machines and have the program calculate them - Uses breakpoints.txt to determine sensitive/resistant/intermediate status based on the MIC. If no information about the organism-drug combo’s available in breakpoints.txt, the program will print a warning and set the call to "X".

The dropdown menu allows you to select which parser you want to use.

  • Vitek1 - A parser for a text report from the bioMeriux VITEK2 machine (format 1).
  • Phoenix1 - A parser for a text report from the BD Phoenix machine (format 1).
  • Custom parser - A parser that reads in a custom parser file and uses it to determine how to parse the input file.

Once you’ve selected your options, hit "OK" to continue, "Cancel" to close the program, or "About" to see the program’s license.

After hitting "OK", select your input file. After the program calculates the necessary info, it’ll either automatically write your files to the default locations specified in config.txt, or it’ll prompt you to give it a place to put them.

6.1 Warnings

Sometimes, information that the program thinks should be in its data files won’t be there. When that happens, it’ll output a warning to warnings.txt, found in the same directory as the program. Here are examples: 

Warning: the organism name "Staphylococcus aureus" does not appear in
organisms.txt. Either this is a typo, or the name is missing from the file.

No breakpoint found for Klebsiella pneumoniae and Amikacin

When the program tells you that things are missing, add them, and shoot me an email so that I can add them to the data files for the next version of the program.

6.2 Crashes

Sometimes, the program will close unexpectedly. When this happens, here’s what to do.

  • Is there a "lib" folder in the same directory as MDRevealer.jar, and does that folder have a file called "libintl.jar" in it?
  • Are you using the right parser? Using, say, the Vitek parser on a file from a Phoenix machine might cause unexpected behavior.
  • If you’re sure that’s not the case, look in warnings.txt. At the bottom, you should see something like: 
    Fatal error: class java.lang.IndexOutOfBoundsException

java.util.ArrayList.rangeCheck(ArrayList.java:635)
java.util.ArrayList.get(ArrayList.java:411)
MDRevealer.Output.printOutput(Output.java:91)
MDRevealer.Parser.parse(Parser.java:403)
MDRevealer.MDRevealer$ButtonListener.actionPerformed(MDRevealer.java:188)

    Send your config file, your warnings file, and an anonymized version of your input file to me, and I’ll fix whatever the problem is.

Appendix B Contributing

Stuff that needs to be done:

  • - Translations! Right now, the program’s English-only, and if you know another language and are willing to contribute, please do so. Any language will do- Spanish, Japanese, Welsh, Navajo, Esperanto, Klingon... Just download poedit (poedit.net), rename MDRevealer.po to your language’s ISO 631 code (en for English, fr for French, ar for Arabic, zh_TW for Traditional Chinese, etc.), and translate whatever strings you can.

B.1 Using gettext

To extract strings for translation, run this command in MDRevealer/src/MDRevealer (with GNU gettext installed): 

xgettext -k_ --add-comments="TRANSLATORS:" -o ../../po/[lang].po MDRevealer.java
call.java mtf.java Resistances.java Output.java Parser.java Organism.java

Where [lang] is your ISO 631 language code.

So for French, you would run: 

xgettext -k_ --add-comments="TRANSLATORS:" -o ../../po/fr.po MDRevealer.java
call.java mtf.java Resistances.java Output.java Parser.java Organism.java

You can also run makemdrevealer.sh, and rename po/MDRevealer.po to fr.po or whatever.

To compile the program, add this line to makemdrevealer.sh: 

msgfmt --java2 -d build/classes -r MDRevealer -l [lang] po/[lang].po

Again, [lang] is your ISO 631 language code.

So for French, you would add:

msgfmt –java2 -d /build/classes -r MDRevealer -l fr po/fr.po

Then run the script. See Compiling.

Appendix C Compiling

makemdrevealer.sh is a script to automatically compile the program, add any .po files, generate a default .po file, compile this manual, and package a .zip file for release, as well as another .zip with the source.

To run it, you will need:

  • gettext (to extract the strings)
  • texinfo (to build the manual)
  • zip (to make the zips)
  • apache-ant (to compile)
  • java-environment (to compile... and to run the program, obviously!)

Note that these package names only apply to Parabola GNU/Linux and other Arch Linux forks (as well as Arch itself). Other Unix-like operating systems may use different names for them.

If you’re on a Unix-like system (GNU/Linux, *BSD, OS X, etc.), just run this from a shell: ./makemdrevealer.sh. If you’re on Windows, as RMS says, "Either throw Windows out of your computer or throw your computer out the window" ;)


If you aren’t willing to do that, just download Cygwin and run ./makemdrevealer.sh.

Before running it, be sure to set "version" and "revision" to the current version and revision numbers, respectively!

Also note that the script will overwrite data/config.txt with defaultconfig.txt. To prevent this, place a # in front of: 

cp -f defaultconfig.txt data/config.txt
